## R>eYMeN — Session / Database / Hafıza Yapısı Özeti

### 1. `session_db.py` (700 satır) — Advanced Session Storage

**Ne işe yarar:** SQLite WAL + FTS5 tabanlı, Hermes Agent seviyesinde oturum (session) yönetimi. ~30 sütunlu `sessions` tablosu ile her oturumu kaydeder.

**İçerdiği tablolar:**
- **`sessions`** (ana tablo): session_id, source, user_id, model, model_config, system_prompt, parent_session_id, started_at, ended_at, end_reason, message_count, tool_call_count, input/output/cache_read/cache_write/reasoning_tokens, billing bilgileri, cost tahmini/gerçek, title, api_call_count → **30+ sütun**
- **`session_messages`**: session_id, rol, icerik, created_at
- **`session_tool_calls`**: session_id, tool_name, args, result, duration_ms
- **`ajan_gunlugu`** (FTS5, geriye uyumluluk): hedef, eylem, sonuc
- **`ajan_gunlugu_trigram`** (FTS5 trigram, opsiyonel — eski SQLite'da çalışmaz)
- **`session_messages_fts`** (FTS5, content-sync): session_id UNINDEXED, rol UNINDEXED, icerik
- **`session_messages_trigram`** (FTS5 trigram, partial search)

**FTS5 kullanımı:** Evet — 4 adet FTS5 virtual table:
- `ajan_gunlugu` (standart tokenizer)
- `ajan_gunlugu_trigram` (trigram tokenizer — substring arama)
- `session_messages_fts` (content-sync, standart tokenizer)
- `session_messages_trigram` (trigram tokenizer)

**Hafıza budama (consolidation):** Yok. Session bazlı çalışır, otomatik temizleme veya eski kayıt budama yok.

**Eksik/bozuk kısımlar:**
- Thread-safe ama her işlem için bağlantı aç/kapa yapıyor (performans darboğazı olabilir)
- `session_ara()` FTS5 fallback'i LIKE kullanıyor, trigram için fallback yok
- Otomatik temizleme/purge mekanizması yok

**Veritabanı konumu:** `.ReYMeN/session.db` (varsayılan)

### 2. `memory_agent.py` (257 satır) — Conversation Memory Agent

**Ne işe yarar:** Sliding-window context yönetimi + otomatik özetleme + JSON kalıcı kayıt. LLM ile eski mesajları özetleyerek context'te tutar.

**Mimari:**
- `context_length` (varsayılan 50 mesaj) aşılınca en eski `summarize_after` (30) mesajı özetler
- Özet `long_term_summary`'e eklenir, eski mesajlar atılır
- Opsiyonel ChromaDB vektör hafıza desteği (`use_vector_memory=True`)
- JSON dosyaya kayıt: `.ReYMeN/hafiza.json`
- Singleton pattern (`hafiza_kur()`, `hafiza_sifirla()`)

**FTS5 ile ilgili:** Hiç yok. Tamamen JSON + RAM tabanlı.

**Hafıza budama:** Var (sliding window). `_summarize_oldest()` metodu eski mesajları LLM ile özetleyip atar. LLM yoksa direkt keser (en eski mesajları siler).

**Eksik/bozuk kısımlar:**
- ChromaDB sadece `use_vector_memory=True` ile çalışır, varsayılan kapalı
- LLM yokken özetleme yapamaz, direkt atar (bilgi kaybı)
- Thread-safe değil

### 3. `hafiza_genislet.py` (627 satır) — Gelişmiş Hafıza Sistemi

**Ne işe yarar:** Hermes Agent'in FTS5 tabanlı session hafızasının ReYMeN uyarlaması. Koleksiyon bazlı kayıt, FTS5 tam metin arama, kullanıcı tercihleri, cross-session arama, TTL desteği.

**İçerdiği tablolar (`.reymen_hafiza/hafiza.db`):**
- **`kayitlar`**: id, session_id, koleksiyon, anahtar, icerik, metadata (JSON), zaman, expire_zaman
- **`sessions`**: id, baslik, baslangic, bitis, mesaj_sayisi, ozet
- **`tercihler`**: anahtar (PK), deger, guncelleme_zamani (key-value)
- **`kayitlar_fts`** (FTS5, content-sync + trigger'lar): icerik, metadata, anahtar — `unicode61` tokenizer

**Koleksiyonlar:** `konusmalar`, `notlar`, `kullanici_tercihleri`, `session_ozetleri`

**FTS5 kullanımı:** Evet — `kayitlar_fts` FTS5 tablosu, content-sync trigger'lar ile (INSERT/DELETE/UPDATE trigger'ları). Ayrıca FTS5 çalışmazsa LIKE fallback.

**Hafıza budama (consolidation):**
- **TTL Sistemi**: `kaydet()` fonksiyonunda `ttl_saat` parametresi (0=sonsuz, >0 saat bazında expire)
- **`temizle()` metodu**: expire_zaman < eski kayıtları siler, varsayılan 72 saat
- **`_checkpoint()` metodu**: Her 10 mesajda WAL checkpoint

**Eksik/bozuk kısımlar:**
- Singleton pattern class seviyesinde değil, modül sonunda `hafiza = GelismisHafiza()` ile
- `_koleksiyon_id()` metodu boş (return 0) — eski kod kalıntısı
- SQLite bulunamazsa sessizce devre dışı kalır
- Aynı anda `session_db.py` ile yarışan iki farklı hafıza sistemi var

### 4. `bounded_memory.py` (317 satır) — Sınırlı Bellek (LRU)

**Ne işe yarar:** LRU (Least Recently Used) mantığıyla çalışan, maksimum kapasiteli basit bir in-memory key-value deposu. Kapasite dolunca en eski öğeyi atar.

**Mimari:**
- `OrderedDict` tabanlı LRU
- Maksimum boyut (`max_boyut`, varsayılan 100)
- JSON otomatik kaydetme (opsiyonel)
- Zaman damgası takibi
- İstatistik toplama (ekleme, silme, hit, miss, atılma)

**FTS5 ile ilgili:** Hiç yok.

**Hafıza budama:** Evet — LRU budama (`_en_eskiyi_at()`). Kapasite dolunca en eski (en az kullanılan) öğeleri otomatik siler.

**Eksik/bozuk kısımlar:**
- Çok basit, sadece in-memory + JSON persist
- Özetleme veya smart consolidation yok
- DIŞ dosyaya kaydetme (`kaydet()` metodu) MEMORY.md formatında append yapar
